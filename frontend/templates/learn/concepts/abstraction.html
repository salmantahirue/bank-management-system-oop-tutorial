👻 Abstraction

Hiding complexity and defining a contract that subclasses must follow.

What is Abstraction?

Abstraction focuses on what an object does rather than how it does it. In Python, we use Abstract Base Classes (ABC) to define a contract that subclasses must implement.

Real-World Analogy: A "Vehicle" is an abstraction. You can't create a generic "Vehicle" object—you must create a specific type like "Car" or "Bike" that implements the vehicle's methods.

Abstract Base Class in Our Project

The Account class is an Abstract Base Class. You cannot create an Account object directly—you must create a SavingsAccount or CurrentAccount.

📄 backend/domain/account.py 
from abc import ABC, abstractmethod

class Account(ABC):
    """
    Abstract Base Class: Cannot be instantiated directly.
    It defines a CONTRACT that all account types must follow.
    """
    
    def __init__(self, account_id: str, user_id: str, balance: int):
        self.account_id = account_id
        self.user_id = user_id
        self._balance = balance
    
    def deposit(self, amount: int) -> int:
        """Concrete method: shared by all subclasses"""
        if amount <= 0:
            raise ValidationError("Amount must be positive")
        self._balance += amount
        return self._balance
    
    @abstractmethod
    def withdraw(self, amount: int) -> int:
        """
        Abstract method: MUST be implemented by subclasses.
        This is the CONTRACT.
        """
        pass  # No implementation here!
    
    @property
    @abstractmethod
    def account_type(self) -> str:
        """Abstract property: MUST be implemented by subclasses"""
        pass

What Happens If You Try to Create an Account?

# ❌ This will FAIL:
account = Account("acc-123", "user-456", 1000)
# TypeError: Can't instantiate abstract class Account 
# with abstract methods withdraw, account_type

Python enforces the contract: You must implement all abstract methods before you can create an instance.

Why Use Abstraction?

1. Enforces a Contract

By making withdraw() abstract, we guarantee that every account type must implement it. You can't forget to add withdrawal logic.

2. Hides Implementation Details

Code that uses Account doesn't need to know if it's a Savings or Current account. It just calls withdraw() and trusts that the subclass handles it correctly.

3. Prevents Invalid Objects

You can't accidentally create a generic "Account" that doesn't have withdrawal rules. You must choose a specific type.

Abstract vs Concrete Methods

Type Method Purpose
Concrete deposit() Has implementation. All subclasses inherit it as-is.
Abstract withdraw() No implementation. Subclasses MUST provide their own.
Abstract account_type No implementation. Subclasses MUST return "savings" or "current".

How Subclasses Implement the Contract

SavingsAccount Implementation

class SavingsAccount(Account):
    def withdraw(self, amount: int) -> int:
        # ✅ Implements abstract method
        if amount > self._balance:
            raise InsufficientFundsError(...)
        self._balance -= amount
        return self._balance
    
    @property
    def account_type(self) -> str:
        # ✅ Implements abstract property
        return "savings"

CurrentAccount Implementation

class CurrentAccount(Account):
    def withdraw(self, amount: int) -> int:
        # ✅ Implements abstract method (differently!)
        new_balance = self._balance - amount
        if new_balance < -self.OVERDRAFT_LIMIT:
            raise InsufficientFundsError(...)
        self._balance = new_balance
        return self._balance
    
    @property
    def account_type(self) -> str:
        # ✅ Implements abstract property
        return "current"

🎯 Key Takeaway

Abstraction defines what must be done (the contract) without specifying how it's done (the implementation). This:

  • ✅ Ensures all account types follow the same interface
  • ✅ Prevents creating incomplete or invalid objects
  • ✅ Makes the code more maintainable and predictable
