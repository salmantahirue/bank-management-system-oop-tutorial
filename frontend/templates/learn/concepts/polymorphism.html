🎭 Polymorphism

Same interface, different behavior based on the object's type.

What is Polymorphism?

Polymorphism means "many forms." In OOP, it allows objects of different classes to be treated through the same interface, but each class implements the behavior differently.

Real-World Analogy: Both a car and a bicycle have a "move()" method, but they move differently. You can call vehicle.move() without knowing if it's a car or bicycle.

Polymorphism in Our Bank System

Both SavingsAccount and CurrentAccount have a withdraw() method, but they implement it differently:

SavingsAccount.withdraw()

📄 backend/domain/savings_account.py 
def withdraw(self, amount: int) -> int:
    self._require_positive_amount(amount)
    
    # ❌ NO OVERDRAFT: balance must stay ≥ 0
    if amount > self._balance:
        raise InsufficientFundsError(
            "Savings account cannot go below 0."
        )
    
    self._balance -= amount
    return self._balance

CurrentAccount.withdraw()

📄 backend/domain/current_account.py 
def withdraw(self, amount: int) -> int:
    self._require_positive_amount(amount)
    
    # ✅ OVERDRAFT ALLOWED: balance can go to -500
    new_balance = self._balance - amount
    if new_balance < -self.OVERDRAFT_LIMIT:
        raise InsufficientFundsError(
            "Overdraft limit exceeded."
        )
    
    self._balance = new_balance
    return self._balance

Polymorphism in Action: Service Layer

Look how the AccountService uses polymorphism without knowing the account type:

📄 backend/services/account_service.py 
def withdraw(self, user_id: str, amount: int) -> int:
    account = self._get_account(user_id)
    
    # 🎭 POLYMORPHISM: This works for BOTH SavingsAccount and CurrentAccount!
    # Python automatically calls the correct withdraw() method
    new_balance = account.withdraw(amount)
    
    # The service doesn't need to know if it's Savings or Current!
    # Each account type handles its own rules.
    
    self.account_repo.save(account)
    return new_balance

Why Polymorphism is Powerful

❌ Without Polymorphism

def withdraw(self, account, amount):
    if account.type == "savings":
        # Savings logic here
    elif account.type == "current":
        # Current logic here
    # 😰 What if we add a new account type?
    # We have to modify THIS code!

Problem: Tightly coupled, hard to extend.

✅ With Polymorphism

def withdraw(self, account, amount):
    # 🎯 Just call withdraw() - each class handles itself!
    return account.withdraw(amount)
    # ✅ Adding a new account type? 
    # Just create the class, no changes here!

Benefit: Loose coupling, easy to extend.

🎯 Key Takeaway

Polymorphism lets you write code that works with the interface (the method name) rather than the implementation (the specific class). This makes your code:

  • ✅ More flexible (easy to add new account types)
  • ✅ More maintainable (changes are isolated to one class)
  • ✅ More readable (the intent is clear: "withdraw money")
