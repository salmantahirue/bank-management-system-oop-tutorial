🔒 Encapsulation

Hiding internal state and controlling access through methods.

What is Encapsulation?

Encapsulation is the practice of hiding internal details of a class and providing controlled access through public methods. In Python, we use naming conventions and properties to achieve this.

Real-World Analogy: Think of a car. You don't directly access the engine's internal components. Instead, you use the steering wheel, pedals, and buttons (public interface) to control it.

Where We Use It: The Account Class

In our Account class, we protect the balance from direct modification:

📄 backend/domain/account.py 
class Account(ABC):
    def __init__(self, account_id: str, user_id: str, balance: int):
        self.account_id = account_id
        self.user_id = user_id
        # ⚠️ Protected attribute: starts with underscore
        self._balance = balance  # Cannot be accessed directly from outside
    
    @property
    def balance(self) -> int:
        """Read-only access to balance"""
        return self._balance
    
    def deposit(self, amount: int) -> int:
        """✅ Controlled way to modify balance"""
        if amount <= 0:
            raise ValidationError("Amount must be positive")
        self._balance += amount
        return self._balance

Why Use Encapsulation Here?

❌ Without Encapsulation

# Anyone can do this:
account.balance = -1000  # 💥 Negative balance!
account.balance = "invalid"  # 💥 Wrong type!
account.balance += 1000000  # 💥 No validation!

Problem: No validation, no control, easy to break the system.

✅ With Encapsulation

# Must use methods:
account.deposit(100)  # ✅ Validated
account.withdraw(50)  # ✅ Validated
balance = account.balance  # ✅ Read-only access

Benefit: All changes go through validation. Balance stays consistent.

Python Naming Conventions

Naming Meaning Example
balance Public attribute Anyone can read/write
_balance Protected attribute Convention: "Don't access directly"
__balance Private attribute Name mangling (rarely used)

🎯 Key Takeaway

Encapsulation protects your data integrity. By forcing all balance changes through deposit() and withdraw(), we ensure:

  • ✅ Amounts are always validated (positive, sufficient funds)
  • ✅ Business rules are enforced (no negative balance for savings)
  • ✅ Future changes are easier (modify one method, not everywhere)
