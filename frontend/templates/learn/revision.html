Revision & Quick Reference

Quick summaries, key takeaways, and common mistakes to avoid.

OOP Concepts Summary

🔒 Encapsulation

What:
Hiding internal state and controlling access
How:
Use protected attributes (_balance) and methods (deposit, withdraw)
Why:
Protects data integrity and enforces validation
In Our Project:
Account._balance is protected, accessed only through methods

🧬 Inheritance

What:
Child classes inherit from parent classes
How:
class SavingsAccount(Account)
Why:
Code reuse and consistency
In Our Project:
SavingsAccount and CurrentAccount inherit from Account

🎭 Polymorphism

What:
Same method name, different behavior
How:
Override methods in subclasses
Why:
Flexible, extensible code
In Our Project:
withdraw() behaves differently for Savings vs Current

👻 Abstraction

What:
Define contract without implementation
How:
Abstract Base Classes (ABC) with @abstractmethod
Why:
Enforces interface and prevents invalid objects
In Our Project:
Account is abstract, cannot be instantiated directly

Key Takeaways

  • Think in Objects: Identify real-world entities (User, Account, Transaction) and model them as classes
  • Separation of Concerns: Domain classes (what), Services (how), Repositories (storage)
  • Encapsulation First: Always protect your data. Don't expose internal state directly
  • Inheritance for "Is-A": Use when you have a true "is-a" relationship (SavingsAccount IS AN Account)
  • Composition for "Has-A": Use when one class uses another (Service HAS repositories)
  • Abstract for Contracts: Use ABC when you want to enforce that subclasses implement certain methods
  • Polymorphism for Flexibility: Same interface, different implementations make code extensible

Common Mistakes to Avoid

❌ Mistake: Exposing Internal State

class Account:
    def __init__(self, balance):
        self.balance = balance  # Anyone can modify!

✅ Fix: Use protected attribute and methods

class Account:
    def __init__(self, balance):
        self._balance = balance
    def deposit(self, amount):
        self._balance += amount

❌ Mistake: Using Inheritance for Everything

class AccountService(AccountRepository):  # Wrong!

✅ Fix: Use composition

class AccountService:
    def __init__(self):
        self.repo = AccountRepository()  # Has-a, not is-a

❌ Mistake: No Validation

def deposit(self, amount):
    self.balance += amount  # No checks!

✅ Fix: Always validate

def deposit(self, amount):
    if amount <= 0:
        raise ValidationError("Amount must be positive")
    self._balance += amount

❌ Mistake: Mixing Concerns

class Account:
    def save_to_file(self):  # Storage in domain class!
        # File I/O here
        pass

✅ Fix: Separate layers

# Domain class - no file I/O
class Account:
    def deposit(self, amount): pass

# Repository - handles storage
class AccountRepository:
    def save(self, account): pass

Interview Quick Tips

When Asked About Encapsulation

Say: "Encapsulation protects data by hiding internal state. In our project, _balance is protected, and we use deposit() and withdraw() methods to modify it, ensuring validation."

When Asked About Inheritance

Say: "Inheritance allows code reuse. SavingsAccount and CurrentAccount inherit from Account, getting shared attributes and methods, but each implements withdraw() differently."

When Asked About Polymorphism

Say: "Polymorphism means same interface, different behavior. Both account types have withdraw(), but Savings prevents overdraft while Current allows it up to -500."

When Asked About Abstraction

Say: "Abstraction defines a contract. Account is abstract - you can't create it directly. You must create SavingsAccount or CurrentAccount that implement all abstract methods."
