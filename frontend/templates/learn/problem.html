The Problem Statement

Understanding the real-world problem we're solving with Object-Oriented Programming.

📋 The Real-World Scenario

Imagine you're a software developer at a startup bank. Your manager asks you to build a Bank Management System that allows customers to:

  • Create accounts and register for online banking
  • Deposit money into their accounts
  • Withdraw money (with different rules for different account types)
  • View their transaction history
  • Check their current balance

The system must be reliable (data shouldn't disappear), secure (users must log in), and extensible (easy to add new features later).

🎯 Why This Problem is Perfect for Learning OOP

This isn't just a coding exercise—it's a real-world problem that naturally requires OOP concepts:

1. Multiple Entity Types

We have Users, Accounts, and Transactions—each is a distinct "thing" that needs its own class.

2. Different Account Behaviors

Savings and Current accounts have different withdrawal rules—perfect for inheritance and polymorphism.

3. Data Protection

Balance shouldn't be changed directly—encapsulation protects it.

4. Future Extensibility

We might add "PremiumAccount" or "BusinessAccount" later—OOP makes this easy.

📝 Detailed Requirements

1. User Management

  • Registration: Users can create an account with a username and password.
    • Username must be unique
    • Password must be at least 4 characters (for learning purposes)
    • Password should be hashed before storage
  • Authentication: Users can log in with their credentials.
    • System validates username and password
    • Returns a session token (simplified for learning)

2. Account Management

  • Account Creation: Each user gets one account when they register.
    • User chooses account type: Savings or Current
    • Initial balance is 0
    • Account ID is auto-generated
  • Account Types:
    • Savings Account: Cannot go below 0 (no overdraft)
    • Current Account: Can go down to -500 (overdraft limit)
  • Balance Inquiry: Users can check their current balance at any time.

3. Transaction Operations

  • Deposit:
    • Amount must be positive
    • Balance increases by the deposit amount
    • Transaction is recorded in history
  • Withdraw:
    • Amount must be positive
    • Savings: Balance cannot go below 0
    • Current: Balance cannot go below -500
    • Transaction is recorded in history
    • Error if insufficient funds

4. Transaction History

  • Every deposit and withdrawal is recorded
  • Each transaction stores: timestamp, type (DEPOSIT/WITHDRAW), amount, optional note
  • Users can view their complete transaction history
  • Transactions are immutable (cannot be edited or deleted)

5. Data Persistence

  • All data is stored in a JSON file (data/bank_data.json)
  • No database required (simplified for learning)
  • Data persists across server restarts
  • File-based storage is easy to inspect and debug

🧠 The "Intern" vs. "Architect" Mindset

❌ The Beginner Way

Write one big main.py file.

Use global variables for data.

Put logic inside if/else blocks.

✅ The Architect Way

Identify Entities (User, Account).

Define Responsibilities (Who handles login? Who handles storage?).

Use Classes to encapsulate data and behavior.
